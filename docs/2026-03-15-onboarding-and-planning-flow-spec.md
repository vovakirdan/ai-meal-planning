# Onboarding And Planning Flow Spec

**Date:** 2026-03-15  
**Status:** Agreed draft  
**Scope:** onboarding, weekly planning flow, dish modeling constraints for MVP

## Purpose

Этот документ фиксирует уже согласованные продуктовые решения, чтобы не потерять их перед проектированием схемы БД и реализацией flow.

## Core Product Assumptions

- MVP строится как household meal planner в Telegram.
- В рамках одного household все едят один и тот же набор блюд.
- Раздельная готовка не поддерживается.
- Участники household нужны для учета пересечения вкусов, ограничений и персонального feedback.
- Reminder-настройки живут на уровне `telegram_user`.
- Pantry живет на уровне household.

## Onboarding

### Entry Point

- `/start` запускает guided Q&A onboarding flow.
- Onboarding должен ощущаться как короткий бытовой диалог, а не как длинная анкета.
- Все ответы должны редактироваться позже без пересоздания household.

### Goal

Собрать минимально достаточный household context, чтобы после onboarding сразу предложить пользователю составить первый недельный план.

### Household-Level Questions

На уровне household бот собирает:

- количество человек в household;
- сколько приемов пищи в день обычно планировать;
- включать ли десерты;
- `repeatability_mode`.

### Meal Template Rules

- Пользователь выбирает количество приемов пищи в день в диапазоне `2-5`.
- По умолчанию используется `3`.
- Отдельно задается флаг включения десертов.
- В UI пользователь выбирает именно количество приемов пищи, а не список слотов.
- Внутри системы это разворачивается в фиксированные слоты.

Внутреннее отображение:

- `2` = `breakfast`, `dinner`
- `3` = `breakfast`, `lunch`, `dinner`
- `4` = `breakfast`, `lunch`, `dinner`, `snack_1`
- `5` = `breakfast`, `lunch`, `dinner`, `snack_1`, `snack_2`
- `dessert` идет отдельным флагом поверх этого

### Repeatability Mode

- Поле можно пропустить.
- Значение по умолчанию: `balanced`.

### Household Members

После household-level вопросов бот отдельно проходит анкету по каждому участнику household.

Для каждого участника обязательны:

- имя/ярлык;
- ограничения;
- любимые кухни;
- свободная заметка `что важно учитывать`.

### Explicit Non-Field: Dislikes

- `dislikes` не собираются как отдельное профильное поле на onboarding.
- Негативная реакция на блюдо живет на уровне dish feedback.
- Если пользователь в диалоге говорит что-то вроде `не люблю оливки`, это может быть интерпретировано как обновление профильных ограничений или предпочтений, а не как comment на конкретное блюдо.

### Reminders During Onboarding

Reminder-настройки собираются прямо в onboarding.

Настраиваются:

- ежедневный reminder на оценку блюд: время;
- weekly reminder на составление следующего плана: день недели и время.

Обе настройки хранятся на уровне `telegram_user`.

### Pantry Step

- В конце onboarding бот предлагает опционально заполнить pantry.
- Pantry не является обязательной частью onboarding.
- После отказа onboarding все равно считается завершенным.

Поддерживаемые состояния pantry для MVP:

- `есть`;
- `мало`;
- `нет`;
- опционально `меньше X грамм / мл / штук`.

### Onboarding Exit

После завершения onboarding бот сразу предлагает перейти к `plan_week`.

## Weekly Planning Flow

### Default Planning Logic

- План всегда строится на явный диапазон дат.
- Таймзона по умолчанию: `Europe/Moscow`.
- Если пользователь запускает планирование в середине недели, бот по умолчанию предлагает следующую полную неделю.
- Полная неделя определяется как `понедельник-воскресенье`.
- Альтернативно пользователь может начать план с сегодня, с ближайшего дня или с произвольной даты.

### Week-Level Questions Before Generation

Перед генерацией бот уточняет:

- диапазон дат;
- оставить ли текущий household meal template;
- настроение недели;
- есть ли специальные пожелания или ограничения именно на эту неделю;
- нужно ли учитывать pantry.

Обязательные поля перед генерацией:

- диапазон дат;
- подтверждение meal template на неделю.

Все остальные вопросы можно пропустить.

### Week Mood

- `настроение недели` относится только к конкретному `plan_week`.
- Это не поле профиля.
- Значение выбирается из предустановленных вариантов.
- Поле можно пропустить.
- Если поле пропущено, бот не делает явный уклон в одну кухню или стиль.

### Draft Generation

- После краткого обсуждения weekly context бот берет паузу на генерацию.
- Затем присылает draft меню на весь выбранный диапазон.
- Draft должен быть сохранен в БД сразу после генерации.
- Причина: пользователь должен мочь вернуться к draft после проблем с сетью или ошибки.

### Plan Structure

- Weekly plan household-level, не персональный.
- `planned_meal` это прием пищи.
- Один прием пищи может содержать несколько блюд.

### Editing Flow

Редактирование происходит иерархически:

- день;
- прием пищи;
- блюдо.

На уровне блюда бот сразу показывает:

- рецепт;
- ингредиенты.

Быстрые действия кнопками:

- `сменить`;
- `сделать легче / меньше`;
- `лайк`;
- `исключить навсегда`.

Свободный текст используется для тонких правок, например:

- `добавь больше пармезана`;
- `подбери аналог без сахара`;
- `сделай неострым`.

### Local Rebuild Rules

- Если пользователь редактирует конкретное блюдо, по умолчанию пересобирается только это блюдо.
- Локальная правка блюда не должна автоматически пересчитывать баланс дня.
- Более широкая пересборка должна происходить только по явному действию пользователя.

### Plan Confirmation

- После локальных правок пользователь подтверждает неделю целиком.
- Подтверждение не делится по дням.

## Dish Modeling

### Dish Identity

- Каноническое блюдо хранится как отдельная сущность `dish`.
- Ингредиенты тоже хранятся как отдельные сущности `ingredient`.
- После подтверждения блюда в план вставляется `dish_id`.
- К `dish_id` привязываются recipe, feedback, history и ingredient relations.

### Draft vs Confirmed

- В draft храним snapshot предложенного блюда.
- Snapshot включает название, рецепт, ингредиенты и adaptation notes.
- В draft не обязателен канонический `dish_id`.
- Канонизация происходит при подтверждении.

### Canonical Matching

- При подтверждении система должна пытаться переиспользовать существующий `dish_id`.
- Новый `dish` создается только если это действительно новое блюдо.

### What Counts As The Same Dish

- Небольшие адаптации рецепта не создают новый `dish`.
- Изменение количества ингредиента, жирности, остроты, масла и похожих параметров остается в рамках одного `dish_id`.
- Замена ключевой основы блюда, например `кальмар -> курица`, считается новым блюдом и требует нового `dish_id`.

### Profile-Specific Adaptation

- Канонический рецепт не должен адаптироваться глобально под всех.
- Адаптация выполняется под конкретный household/member context во время планирования.
- Основа для адаптации: канонический рецепт + профильный опыт по блюду.
- Если профильного опыта нет, используется канонический рецепт как есть.

### Adaptation Notes

Если блюдо адаптировано под household context, это должно храниться как adaptation notes рядом с использованием блюда в плане, например:

- `убрать оливки`;
- `меньше масла`;
- `неострое`.

Если пользователь сообщает об общей непереносимости или нелюбви к ингредиенту в диалоге планирования, это может обновить профильные ограничения, а не feedback на конкретный dish.

## Feedback

### Scope

- Feedback по блюдам собирается отдельно по каждому участнику household.
- Жесткие исключения уровня `never_again` действуют на весь household.

### Daily Feedback Reminder

- Если на день был план, бот может в выбранное время прислать сообщение в духе `Давай оценим блюда`.
- Feedback flow идет по схеме `блюдо -> участники household`.

### Feedback Form

Для каждого блюда:

- сначала быстрый verdict;
- потом опциональный свободный комментарий.

Свободный текст может содержать сигналы вроде:

- `слишком жирно`;
- `слишком остро`;
- `слишком долго`.

Дальше ИИ может нормализовать это в полезные adaptation notes для будущего планирования.

## Pantry

- Pantry один общий на весь household.
- Пользователь может в любой момент вручную:
  - добавить продукт;
  - скорректировать остаток;
  - удалить продукт из pantry.
